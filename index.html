      Scalable Vector Graphics (SVG) is an XML markup language for describing two-dimensional vector graphics. 
    

    

      What that means to us front-end developer folks is that we can write code similar to what XML looks like you might have seen before to describe how our SVG looks.

    

    

      What that means to us designerey folks is that since it's vector-based, our images can scale up or down and not lose their quality or get pixelated looking and blurry.

    

    

      At the most basic code level, SVG's are made of different shapes. They can be circles, rectangles, polygons, paths, lines, or text. 
    


    

      Those are probably the most recognizable elements you'll see if you inspect an SVG. If you check out the mozilla developer docs, you'll find a complete list. https://developer.mozilla.org/en-US/docs/Web/SVG/Element I'm not going to describe every element available but I do think it's worth while to attempt creating your own SVG's and inspecting the code generated by them by opening them up in Sublime or whatever text editor it is that you use.
    


    

      
      How can we create them? Well, you can painstakingly write out all of the code for them yourself, which I don't really recommend unless you're really well versed in how the path notation works. Or you can use a graphics program such as Adobe Illustrator (probably the most expensive option), Sketch (which is my favorite but sadly only avialable for Macs), or Inkscape (which is free and what I use on ubuntu when I'm in that environment). Illustrator is probably the most advanced for creating really complex designs, Sketch is maybe for more simple icon design but also amazing for interface design as a whole, and Inkscape is great if you really like getting into the nitty gritty of SVG code because it has a nice little window that shows your code as you create your shapes and edit them.
    


    

      Really, all that matters is finding a program that works for you and supports the .svg file format.
    



    

      Before we get into how we can use SVGs in our projects, let's talk briefly about browser support. For your most basic SVG use cases, you will be ok unless you have to support IE8, or Android prior to 4.1. There are ways to automate png fallbacks for our svg's that we'll talk about later but if you need to support those browsers, be prepared to do some research and make some decisions  about if you think it's worth the effort. Maybe SVG's aren't the right fit for older projects.
    


    

      At GW, when we were supporting IE8, we were also looking forward. So we use modernizr on almost all of our projects to detect support for SVG. [ show picture of modernizr custom build ] If you create a custom build you can check the svg options under misc. Then you'll get some tools for css and javascript to load in fallbacks or create alternate css.

    


    

      So how can we use SVG's in place of our boring old pngs? Well you can use them as the source of an img tag much like you would any other image. [ show code for using img tag]. no matter how you use an svg, you still get the benefit of it scaling just like you would expect a vector image. [ show example of 3 different img tags with the same svg source and different widths]. Expands just like setting the width and height on an img tag that references a jpg but we don't get the blurry or pixelatedness that we would with other image formats.

      Also, for all of these use cases we'll go over, the same SVG will look just as great on a 4x screen as it will on a non-high resolution screen. We don't need our min-resolution media queries anymore.
    


    

      The code for our SVG's fits right into our markup so you can place all of your SVG code inline. [ show inline svg markup with html]. We can even use our title element for our svg and give it a nice title that will be read when a screenreader tabs to our svg, much like alt text on an image.
    


    

      You'll notice our SVG markup has a width and height attribute set. This is something you'll want to pay attention to because just like setting a fixed width and height on an image, those attribute will restrict our svg from expanding or shrinking in, say, a responsive design. So sometimes, you may want those fixed dimensions, but if we want our SVG to be responsive we can use those dimensions for our viewbox instead. [ show image of dimensions grayed out but new viewbox being used ] We can use those width and height attribute to determine our viewport starting at the top left most part of our svg's "canvas" and drawing a rectangle to the right-most and bottom-most point.
    


    

      Now, in a responsive site, since we don't have those fixed dimensions anymore, we can scale the svg using a percentage.
    


    

      Heres one of those quirks where I said the basic support is there, but you'll still have to do some tweaking for IE in particular. IE will actually collapse all SVG's to be 150px height regardless of their viewbox if there is no width and height attributes set. There is a workaround, though. For Internet Explorer, you'll have to wrap your responsive svg's in another element and use what other's have called the padding-hack. 

      In our markup, create a .svg-wrap div and place your svg inside of it.

      Then show css of padding hack.

      Obviously, you'll have to manage these from image to image if you are using different height svg's all over the place.
    

  
    

      There is a great article with a more thorough explanation I highly recommend reading if you come across issues making your SVG's responsive with a lot of different techniques. [link to codrops] You definitely want to make sure you do some research and know what kind of browser support your project needs.

We use svg sprites for the same reason we used our old bitmap sprites, typically a smaller total combined file size for all of our icons and less http requests which helps with performance.

       Why is this important? ALl that extra text we removed results in smaller file sizes, and we should always be doing these small performance gains just like how we should compress our jpegs and gifs and pngs. There is no reason not to do it. It's osmething we easily have control over.
    


    

      We can do that manually. OR we can get a little help from SVGO.

 So this is where the front-end developer part of me gets excited because now we get to talk performance and automation. SVGO is a node-based command line utility for optimizing SVGs. If you're not familiar with Node, I like to think of it as heavy-lifting javascript. What before we would use back-end languages like PHP or C# for, node let's us do those tasks with javascript. If that sounds scary to you, try not to be scared. Whenever we hire a new front-end developer at gravity works, we always show them the "command line" way first, and it's like an entire new world opens up to them. I've found there is great documentation to go along with all of the node packages I've found, including SVGO.

       Along with node, my team likes to use gulp. Where node does our heavy lifting, gulp let's us create little tasks that we can have our node modules do. You can create one task that does several other tasks and you can pipe them together so they do one task after another. You can watch files for changes to them which is great becuase then those changes can trigger one of our tasks. It's kind of amazing but remember, don't look to task runners or node modules to do BETTER work for you, get them to actually fit into your workflow and make you work more efficiently. Basically don't use a task runner because you think it's the hot new ish, use it because you have pain points in your work flow. Please checkout gulpjs.com if you want a really thorough explanation.
    


    

       By the way, almost all of the node modules I've needed have had a grunt equivalent so don't fret if your team primarily uses Grunt.
